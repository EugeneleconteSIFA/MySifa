Mémo PC local — exemple feature/maintenance
À faire au début

◆ Une seule fois, quand tu démarres la feature

Créer la branche à partir de staging à jour
git checkout staging
git pull origin staging
git checkout -b feature/maintenance
Tu es maintenant sur ta branche perso, isolée de l'autre dev.
À faire en continu

◆ À répéter à chaque cycle de travail

1. Sauvegarder et pousser ton travail sur GitHub
git add .
git commit -m "maintenance: bouton 'Créer une alerte' (placeholder en developpement)"
git push -u origin feature/maintenance
À ce stade la branche est sur GitHub — pas encore sur v1.
2. Déployer sur v1 (merge dans staging)
git checkout staging
git merge feature/maintenance
git push origin staging
Le VPS pull staging chaque minute → v1.mysifa.com se met à jour. Pour retravailler ensuite, git checkout feature/maintenance.

✓ TOUJOURS

  • Une branche par feature, partir de staging à jour
  • Pull staging avant de merger sa propre branche
  • Tester sur v1.mysifa.com avant la prod
  • Promouvoir uniquement via le bouton dans /settings

✗ JAMAIS

  • Push direct sur main
  • Git pull manuel sur le VPS (v1 ou prod)
  • Restart à la main du service mysifa
  • Promouvoir une feature non testée sur v1
